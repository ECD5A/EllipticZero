# Руководство По Вкладу

## Область

Это руководство объясняет, как вносить вклад, не ослабляя базовые
ограничения проекта. Вклад приветствуется, но проект должен оставаться:

- ориентированным на локальный запуск
- изолированным
- воспроизводимым
- опирающимся на доказательную базу
- простым с точки зрения пользователя

Если изменение делает систему шумнее, противоречивее или ближе к
неограниченной исполнительной оболочке, это, скорее всего, неверное
направление.

## Что Прочитать Сначала

Перед изменением кода прочитай:

- `README.md`
- `ARCHITECTURE.md`
- `AGENTS.md`
- `REPRODUCIBILITY.md`
- `REPORT_SPEC.md`
- `SECURITY.md`

## Направление Продукта

Базовый пользовательский цикл должен оставаться простым:

1. выбрать язык
2. выбрать исследовательский домен
3. при необходимости выбрать кривую или передать код контракта
4. ввести свободно сформулированную исследовательскую идею
5. запустить сессию
6. просмотреть доказательную базу и итоговый отчёт

Внутренняя сложность должна оставаться внутри оркестратора, агентного цикла,
политик песочницы, локальных раннеров и слоёв воспроизводимости.

## Приоритетные Направления Вклада

- локальные исследовательские раннеры в песочнице
- ограниченные ECC-тестбеды и синтетические цели
- раннеры, корпуса, repo-casebook-сценарии и качество отчётности для аудита
  смарт-контрактов
- качество multi-agent исследования
- воспроизводимость, трассировки, повторный запуск и манифесты
- качество осторожной отчётности
- ясность документации и установки

Низкоприоритетные направления не должны захватывать roadmap:

- бесконечная косметическая полировка UI
- разрастание второстепенных функций в advanced/internal screens
- спекулятивная логика ранжирования
- всё, что ослабляет границы песочницы

## Правила Безопасности

Пожалуйста, сохраняй эти границы:

- никакого произвольного выполнения из agent prompts
- никакого прямого shell-доступа для reasoning agents
- никаких скрытых remote execution paths
- никакого тихого ослабления трассируемости или возможности повторного запуска
- никаких преувеличенных выводов в отчётах без доказательной базы

Локальные вычисления должны оставаться под контролем утверждённых
инструментов, раннеров и политик.

## Локальная Среда И Проверки

```powershell
python -m venv .venv
.\.venv\Scripts\python.exe -m pip install --upgrade pip
.\.venv\Scripts\python.exe -m pip install -e .[lab]
.\.venv\Scripts\python.exe -m compileall app tests scripts
.\.venv\Scripts\python.exe -m pytest -q
```

На Windows может появляться `PytestCacheWarning`, если `.pytest_cache`
блокируется локальной политикой. Сам по себе этот warning не означает
логическую ошибку.

## Синхронизация Документации

Английская документация является основной.

Обязательная схема English-to-Russian:

- `README.md` -> `README.ru.md`
- `ARCHITECTURE.md` -> `docs/ru/ARCHITECTURE.ru.md`
- `AGENTS.md` -> `docs/ru/AGENTS.ru.md`
- `COMMERCIAL_LICENSE.md` -> `docs/ru/COMMERCIAL_LICENSE.ru.md`
- `REPRODUCIBILITY.md` -> `docs/ru/REPRODUCIBILITY.ru.md`
- `REPORT_SPEC.md` -> `docs/ru/REPORT_SPEC.ru.md`
- `SECURITY.md` -> `docs/ru/SECURITY.ru.md`
- `LICENSE_TRANSITION.md` -> `docs/ru/LICENSE_TRANSITION.ru.md`
- `TRADEMARKS.md` -> `docs/ru/TRADEMARKS.ru.md`
- `CONTRIBUTING.md` -> `docs/ru/CONTRIBUTING.ru.md`

Если один из этих английских файлов меняется, соответствующий русский файл
должен обновляться в той же правке.

## Ожидания От Pull Request

Хорошее изменение обычно включает:

- чёткую формулировку проблемы
- ограниченную реализацию
- тесты
- обновление документации, если меняется пользовательское поведение
- короткое объяснение архитектурного влияния

Когда уместно, отдельно указывай влияние на песочницу, воспроизводимость и
затрагивает ли правка standard mode, exploratory mode или оба режима.

## Лицензирование Вкладов

Перед существенным вкладом сначала открой issue и обсуди предлагаемое
изменение.

Отправляя вклад, ты соглашаешься, что он предоставляется на условиях
лицензии репозитория и что сопровождающий проекта может запросить
дополнительные документы для крупных изменений, если это потребуется для
будущего лицензирования, коммерческого распространения или управления
проектом.

Не отправляй код, если у тебя нет права его лицензировать и передавать.

Крупные вклады должны делаться только по предварительной договорённости с
сопровождающим проекта.
