# Политика Безопасности

## Область

Эта политика описывает вопросы безопасности, наиболее важные для репозитория, особенно если они касаются:

- нарушения границ песочницы
- произвольных или неконтролируемых путей выполнения
- обхода утверждённых границ локальных раннеров и инструментов
- утечек трассировок, пакетов воспроизводимости, данных повторного запуска или артефактов
- ошибок обращения с API-ключами
- проблем воспроизводимости, создающих реальный риск

Этот репозиторий не предназначен для публичной публикации эксплуатационных сценариев.

Небезопасные layout'ы локальных plugin-путей, symlink'и или plugin-файлы вне
корневого каталога не должны загружаться в реестр, а ссылки на артефакты в
manifest и bundle должны фильтроваться, если они разрешаются вне утверждённых
локальных корней хранения. Копии session/trace тоже должны оставаться внутри
этих же корней перед экспортом.

## Как Сообщать

Пожалуйста, не публикуйте неподтверждённые проблемы безопасности как обычные открытые GitHub issue.

Предпочтительный порядок связи:

1. Использовать GitHub private vulnerability reporting, если он включён для репозитория.
2. Если он недоступен, написать сопровождающему проекта на `stelmak159@gmail.com`.
3. Если приватного канала пока нет, открыть минимальное публичное issue с запросом на безопасный способ связи без публикации чувствительных деталей.

## Что Включить В Сообщение

Желательно указать:

- затронутую версию или commit
- операционную систему и версию Python
- точные шаги воспроизведения
- требуется ли нестандартная конфигурация
- затрагивает ли проблема только `mock`, внешние провайдеры или оба режима
- ломает ли проблема границы песочницы или только слой отчётности
- затрагиваются ли локальные артефакты, трассировки, пакеты воспроизводимости или секреты

Если возможно, приложите минимальный воспроизводимый пример, который остаётся в защитных и песочничных рамках репозитория.

## Текущие Границы

EllipticZero намеренно устроен так, что:

- агенты занимаются только рассуждением
- локальное выполнение идёт только через утверждённые раннеры и инструменты
- исследование остаётся локальным и изолированным
- результаты остаются проверяемыми и воспроизводимыми

Сообщения, показывающие нарушение этих границ, особенно ценны.

## Граница Доверия И Данных

EllipticZero разделяет идею пользователя, рассуждения агентов, локальные
инструменты, артефакты и ручную проверку как разные уровни доверия. Вывод
агента является интерпретацией. Доказательную базу несут локальные outputs
инструментов, сохранённые sessions/traces/manifests/bundles и решение
проверяющего.

Провайдер по умолчанию - `mock`. Он детерминированный и не требует API-ключей.
Этот путь нужен для первой оценки проекта, golden cases, проверки документации
и воспроизводимых smoke-тестов.

Hosted providers опциональны. Настраивай `openai`, `openrouter`, `gemini` или
`anthropic` только если осознанно нужен live-ответ модели вместо стандартного
`mock`-пути. Когда используется hosted provider, prompts и ограниченный контекст
агентского вызова могут быть отправлены этому провайдеру по его условиям.

Не используй hosted provider для приватных контрактов, закрытого кода,
клиентских материалов или чувствительных трасс, если такая передача
неприемлема. Команда live-smoke проверяет только связность провайдера; она не
доказывает качество аудита, уровень приватности или production readiness.

Перед live-запуском hosted-провайдера можно посмотреть подготовленный контекст:

```powershell
python -m app.main --provider openrouter --provider-context-preview "Проверить, какой контекст может уйти hosted-провайдеру."
```

Для проверки приватного контракта запускай предварительный просмотр с теми же
contract- и routing-флагами, которые планируешь использовать. Провайдер при
этом не вызывается.

EllipticZero не заявляет изоляцию уровня ОС, containment для malware или
безопасный запуск произвольных недоверенных программ. Враждебные цели нужно
проверять в среде, которой ты уже доверяешь для такой работы.

## Граница Артефактов, SARIF И Заявлений

Сохранённые session и trace snapshots применяют redaction для значений, похожих
на секреты, но всё равно не стоит помещать секреты в prompts, комментарии
контрактов, локальные пути, environment dumps или артефакты.

SARIF export - это мост для CI и Code Scanning при проверке сохранённых запусков:

```powershell
python -m app.main --replay-bundle .\artifacts\bundles\session_id --export-sarif .\artifacts\sarif\session_id.sarif
```

SARIF-записи являются пунктами проверки, а не доказательством уязвимости.
Локальный сигнал не равен автоматически подтверждённой уязвимости, finding card
не является финальным аудиторским вердиктом, а прохождение benchmark не
означает полного покрытия. Если доказательств недостаточно, правильный исход -
ручная проверка, статус `inconclusive` или более узкий follow-up.
